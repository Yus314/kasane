# Migration Guide — Kasane 0.6 → 0.7 (Unreleased)

This guide covers breaking changes after the 0.6.0 release. The dominant
theme is the **R2.x P7+P9 cascade** ([ADR-039](../decisions.md#adr-039-plugin-path-consolidation-r2x)
Plan B execution): the wire-format-aware `WireFace` type leaves the public
plugin API entirely. Plugin code constructs atoms, element styles,
display directives, and ornaments via the post-resolve
[`Style`](https://docs.rs/kasane-core/latest/kasane_core/protocol/struct.Style.html)
type. The `final_*` resolution flags that some `detect_cursors`-style
plugins observed are now read from
[`UnresolvedStyle`](https://docs.rs/kasane-core/latest/kasane_core/protocol/struct.UnresolvedStyle.html)
instead.

> **TL;DR for plugin authors**
>
> 1. `use kasane_core::plugin_prelude::*` no longer brings `WireFace` into
>    scope. Search for `WireFace` and `Face::default()` in your plugin and
>    replace each construction site with a `Style { fg: Brush::*, bg:
>    Brush::*, .. }` literal.
> 2. `Atom::from_wire(face, text)` callers become
>    `Atom::with_style(text, Style { ... })` (note the argument order
>    flip — text first).
> 3. If you observe Kakoune `final_*` resolution flags (cursor detection,
>    bg-fill priority), read them from `atom.unresolved_style().final_fg`
>    / `final_bg` / `final_style`. Constructing `WireFace` directly is no
>    longer possible from plugin crates.
> 4. If you pattern-match `DisplayDirective::StyleInline` / `StyleLine`
>    or `InlineOp::Style`, rename the `face` binding to `style` (the
>    field type also changes from `WireFace` to `Style`).
> 5. If you implement `RenderOrnament` and emit `CursorEffectOrn` /
>    `SurfaceOrn`, rename the `face` field to `style` and pass `Style`
>    instead of `WireFace`.
> 6. If you use `Command::RegisterThemeTokens` (or the
>    `KakouneSafeCommand::register_theme_tokens` builder), the value
>    type changes from `WireFace` to `Style`.

---

## 1. `Element::text` and the prelude

### Before (0.6)

```rust
use kasane_core::plugin_prelude::*;

let label = Element::text("hello", WireFace {
    fg: Color::Named(NamedColor::Cyan),
    ..WireFace::default()
});
let plain = Element::text_with_style("plain", Style::default());
```

### After (0.7)

```rust
use kasane_core::plugin_prelude::*; // no WireFace re-export

let label = Element::text("hello", Style {
    fg: Brush::Named(NamedColor::Cyan),
    ..Style::default()
});
let plain = Element::text("plain", Style::default());
```

`Element::text_with_style` is gone — `Element::text` now takes `Style`
directly. `Color::Named(...)` becomes `Brush::Named(...)`;
`Color::Rgb { r, g, b }` becomes `Brush::rgb(r, g, b)`;
`Color::Default` becomes `Brush::Default`.

## 2. `Atom::from_wire` → `Atom::with_style`

### Before

```rust
let atom = Atom::from_wire(
    WireFace {
        fg: Color::Named(NamedColor::Red),
        attributes: Attributes::BOLD,
        ..WireFace::default()
    },
    "important",
);
```

### After

```rust
let atom = Atom::with_style(
    "important",
    Style {
        fg: Brush::Named(NamedColor::Red),
        font_weight: FontWeight::BOLD,
        ..Style::default()
    },
);
```

Note the argument order flip: `Atom::with_style` takes `(text, style)`,
not `(face, text)`. `Attributes::BOLD` becomes
`font_weight: FontWeight::BOLD`; `Attributes::ITALIC` becomes
`font_slant: FontSlant::Italic`; `Attributes::REVERSE` becomes
`reverse: true`; `Attributes::UNDERLINE` becomes
`underline: Some(TextDecoration { style: DecorationStyle::Solid, color:
Brush::Default, thickness: None })`.

`Atom::from_wire` is now `pub(crate)`. The protocol parser and
`test_support::wire`'s cursor fixtures retain access; plugin code does
not.

## 3. Reading `final_*` resolution flags

Plugins that distinguish "this colour is final, do not inherit" (Kakoune
cursor atoms set `Attributes::FINAL_FG | REVERSE`) used to read those
bits via `WireFace.attributes`. They now read `UnresolvedStyle.final_*`:

### Before

```rust
let face = atom.unresolved_style().to_face();
if face.attributes.contains(Attributes::FINAL_FG) {
    // ... cursor-atom path
}
```

### After

```rust
if atom.unresolved_style().final_fg {
    // ... cursor-atom path
}
```

The semantics are equivalent (both flags map 1:1). The
`UnresolvedStyle` form is preferred because it does not allocate a
WireFace round-trip.

## 4. `DisplayDirective::StyleInline` / `StyleLine` field rename

### Before

```rust
fn display(&self, _state: &AppView<'_>) -> Vec<DisplayDirective> {
    vec![DisplayDirective::StyleLine {
        line: 0,
        face: WireFace {
            bg: Color::Named(NamedColor::Blue),
            ..WireFace::default()
        },
        z_order: 10,
    }]
}
```

### After

```rust
fn display(&self, _state: &AppView<'_>) -> Vec<DisplayDirective> {
    vec![DisplayDirective::StyleLine {
        line: 0,
        style: Style {
            bg: Brush::Named(NamedColor::Blue),
            ..Style::default()
        },
        z_order: 10,
    }]
}
```

Pattern-matching consumers must rename `face` → `style` in the binding
list. The same applies to `DisplayDirective::StyleInline` and to
`InlineOp::Style`.

## 5. `CursorEffectOrn` / `SurfaceOrn` field rename

### Before

```rust
fn render_ornaments(&self, _state: &AppView<'_>, _ctx: &RenderOrnamentContext) -> OrnamentBatch {
    OrnamentBatch {
        cursor_effects: vec![CursorEffectOrn {
            kind: CursorEffect::Halo,
            face: WireFace::default(),
            priority: 10,
            modality: OrnamentModality::Approximate,
        }],
        ..OrnamentBatch::default()
    }
}
```

### After

```rust
fn render_ornaments(&self, _state: &AppView<'_>, _ctx: &RenderOrnamentContext) -> OrnamentBatch {
    OrnamentBatch {
        cursor_effects: vec![CursorEffectOrn {
            kind: CursorEffect::Halo,
            style: Style::default(),
            priority: 10,
            modality: OrnamentModality::Approximate,
        }],
        ..OrnamentBatch::default()
    }
}
```

Field rename `face` → `style`, type `WireFace` → `Style`. Same for
`SurfaceOrn`.

## 6. `Command::RegisterThemeTokens` and `KakouneSafeCommand::register_theme_tokens`

### Before

```rust
let tokens: Vec<(String, WireFace)> = vec![
    ("plugin.token".into(), WireFace {
        fg: Color::Named(NamedColor::Cyan),
        ..WireFace::default()
    }),
];
let cmd = KakouneSafeCommand::register_theme_tokens(tokens);
```

### After

```rust
let tokens: Vec<(String, Style)> = vec![
    ("plugin.token".into(), Style {
        fg: Brush::Named(NamedColor::Cyan),
        ..Style::default()
    }),
];
let cmd = KakouneSafeCommand::register_theme_tokens(tokens);
```

## 7. `ColorResolver` (kasane-gui custom backends)

If your code consumes the GUI `ColorResolver` (custom render backend,
diagnostics overlay, etc.), the WireFace-typed methods are deleted:

| Before                                            | After                                  |
|---------------------------------------------------|----------------------------------------|
| `resolve_face_colors(&WireFace)`                  | `resolve_style_colors(&Style)`         |
| `resolve_face_colors_linear(&WireFace)`           | `resolve_style_colors_linear(&Style)`  |
| `sync_defaults(&WireFace)`                        | `sync_defaults(&Style)`                |

The `Brush::Default` variant replaces `Color::Default` as the inheritance
sentinel; `is_inherit()` semantics are preserved.

## 8. WIT 4.0.0 — `eval-command` at session-ready ([ADR-041](../decisions.md#adr-041-eval-command-in-session-ready-command))

WIT bumps to `kasane:plugin@4.0.0`. The `session-ready-command` variant
gains an `eval-command(string)` case alongside `send-keys`,
`paste-clipboard`, and `plugin-message`:

```wit
variant session-ready-command {
    send-keys(list<string>),
    eval-command(string),    // NEW in 4.0.0
    paste-clipboard,
    plugin-message(message-config),
}
```

Plugins can now issue a Kakoune command body directly at
`on_active_session_ready_effects` without the `<esc>:cmd<ret>`
keystroke-simulation wrapping that `keys::command()` produces. Use it
when the body is more than a few keystrokes — IPC is smaller, no
`<esc>` mode-reset side effect, no per-character escaping.

The variant-case order shifts the discriminant of subsequent cases on
the wire; per [docs/abi-versioning.md Appendix
A](../abi-versioning.md#appendix-a-variant-non-exhaustive-policy)
this is a breaking change requiring a `major.minor` bump, even though
the variant addition is source-compatible in Rust.

**Migration:**

- Bump `abi_version = "4.0.0"` in `kasane-plugin.toml`.
- Pin `kasane-plugin-sdk = "0.7"` (or higher within `0.7.x`).
- `cargo component build` — pure recompile, no source changes
  required.

Opt-in adoption of the new variant is recommended for session-ready
emissions of command bodies; existing `send-keys` paths continue to
work unchanged.

## 8.1 WIT 4.1.0 — `command-error` + `on-command-error-effects` ([ADR-042](../decisions.md#adr-042-command-error-event-via-info_show-marker-attribution) Phase B Step 1)

WIT bumps to `kasane:plugin@4.1.0` adding:

- A new `command-error` record (`plugin-id: string, message: string`)
- A new plugin-api export
  `on-command-error-effects: func(error: command-error) -> runtime-effects;`

Plugins that opt into command-error observability override the export
to surface Kakoune-side command failures (e.g. as a status badge,
info popup, or internal state). The default impl in the SDK macros
returns empty effects — out-of-tree plugins do not need to write the
export to recompile.

Phase B ships in three coordinated steps under the same WIT 4.1.0
bump:

- **Step 1**: WIT wire space reserved + SDK default no-op.
- **Step 2**: host-side dispatch — `state/apply.rs` recognises the
  marker, parses `(plugin-id, message)`, and routes to the originating
  plugin's new export via `PluginRuntime::deliver_command_error_batch`.
- **Step 3**: host auto-wrap — when a plugin sets
  `[handlers] command_error_observability = true` in its manifest,
  every `Command::EvalCommand` emission is automatically wrapped with
  `try %{ <cmd> } catch %{ info -title '__kasane_plugin_error__'
  %{ <plugin-id>%val{error} } }`. Plugin authors write normal
  commands; the host handles attribution end-to-end.

**Migration:**

- Bump `abi_version = "4.1.0"` in `kasane-plugin.toml`.
- Pin `kasane-plugin-sdk = "0.7"` (any 0.7.x).
- Optionally opt into command-error observability by adding
  `[handlers] command_error_observability = true`.
- Optionally override `on-command-error-effects` to surface failures
  programmatically (status badge, info popup, internal state, etc.).
- `cargo component build` — no source changes required.

## 8.2 WIT 4.2.0 — `on-state-changed-tier1-effects` ([ADR-044](../decisions.md#adr-044-handler--effect-tier-hierarchy) Phase B-2, superseded by 8.3)

WIT 4.2.0 added the parallel `on-state-changed-tier1-effects` export so
plugins could opt into the tier-1 contract while the legacy
`on-state-changed-effects` (returning `runtime-effects`) stayed
available. The dual-dispatch path was a transitional shape; ABI 5.0.0
collapses it into the single tier-typed export described in §8.3.

## 8.3 WIT 5.0.0 — tier-typed handler exports ([ADR-044](../decisions.md#adr-044-handler--effect-tier-hierarchy) Phase B-5)

WIT bumps to `kasane:plugin@5.0.0`. The `runtime-effects` record is
removed. The five `runtime-effects`-returning handler exports are
replaced with their ADR-mapped tier:

| Handler | Old return (4.2.0) | New return (5.0.0) | Tier |
|---|---|---|---|
| `on-state-changed-effects` | `runtime-effects` | `kakoune-side-effects` | T1 |
| `on-command-error-effects` | `runtime-effects` | `kakoune-side-effects` | T1 |
| `on-subscription` | `runtime-effects` | `kakoune-side-effects` | T1 |
| `on-io-event-effects` | `runtime-effects` | `process-capable-effects` | T2 |
| `update-effects` | `runtime-effects` | `process-capable-effects` | T2 |

The transitional `on-state-changed-tier1-effects` parallel from 4.2.0
is removed; its role is fulfilled by the renamed
`on-state-changed-effects`. Plugins built against ABI 4.x are
**rejected at load time** with an `AbiCompat::MajorMismatch` diagnostic
pointing here.

### Migration recipe (per plugin)

1. **Bump dependencies and manifest:**
   - `abi_version = "5.0.0"` in `kasane-plugin.toml`.
   - Pin `kasane-plugin-sdk = "0.7"` (any 0.7.x targeting WIT 5.0.0).
2. **Rename the tier-1 state-changed block** (only if you opted into
   the 4.2.0 transitional name):
   ```diff
   - on_state_changed_tier1_effects(dirty) { … }
   + on_state_changed_effects(dirty) { … }
   ```
   The macro hard-rejects the legacy name at expansion with a pointer
   to this section.
3. **Drop the `Effects` alias when you mean tier-1.** `Effects` is now
   `ProcessCapableEffects` (Tier 2). Inside tier-1 handler bodies
   (`on_state_changed_effects`, `on_init_effects`,
   `on_active_session_ready_effects`, `on_command_error_effects`,
   `on_subscription`) return `KakouneSideEffects` directly, e.g.
   ```diff
   - Effects::default()
   + KakouneSideEffects::default()
   ```
   The asymmetric `From<KakouneSideEffects> for ProcessCapableEffects`
   provides the legal lift; the reverse direction is intentionally
   absent so a tier-1 handler cannot widen to spawn a process.
4. **Spawn from a tier-1 handler ⇒ rewrite to tier 2.** If you used
   `Command::SpawnProcess` (or any process-side variant) inside
   `on_state_changed_effects`, the build now fails. Move the spawn to:
   - `handle_key` / `handle_mouse` / `handle_drop` (user-driven Tier 2),
     or
   - chain through pub/sub or `PluginMessage` to a `update_effects`
     arm (command-handler Tier 2).
   The sprout picker rewrite documented in ADR-044 §"External" is the
   canonical example.
5. **`cargo build --target wasm32-wasip2 --release`** — the bindings
   are regenerated from WIT 5.0.0 and the bindgen errors guide any
   remaining type mismatches.

### Helper aliases

- `Effects = ProcessCapableEffects` — Tier 2 alias retained for use in
  `update_effects` / `on_io_event_effects` bodies.
- `effects(Vec<Command>)` — auto-routes each `Command` variant into
  `kakoune_side` or `process_commands`, so existing tier-2 helpers
  (`just_redraw`, `effects_redraw`) keep working.
- `tier1_effects(Vec<KakouneSideCommand>)` — ergonomic Tier 1
  builder; returns `KakouneSideEffects`.
- `kakoune_setup_effects![…]` (and `kakoune_side_setup_effects![…]` as
  alias) — return `KakouneSideEffects`; the SDK supplies
  `From<KakouneSideEffects> for SessionReadyEffects` /
  `BootstrapEffects` so the same body composes inside
  `on_active_session_ready_effects` / `on_init_effects`.

The wire vocabulary that `KakouneSideCommand` admits is fixed by
`kasane-wit/wit/plugin.wit` (§"ADR-044 tier hierarchy"); refer there
when in doubt about which variants are tier-1.

## 9. New opt-in features (no migration required)

These are pure additions that do not affect existing plugin code. Adopt
them incrementally as the feature fits.

### 9.1 Plugin test harness ([Issue #92](https://github.com/Yus314/kasane/issues/92))

The `kasane-plugin-sdk-test` crate ships a mock-host harness so plugins
can be unit-tested natively without compiling to `wasm32-wasip2` or
driving a real Kakoune. Enable it from your plugin crate:

```toml
[features]
test-harness = ["kasane-plugin-sdk/test-harness"]

[lib]
crate-type = ["cdylib", "rlib"]  # rlib is required so the test binary can link
```

```rust
#[cfg(all(test, feature = "test-harness"))]
mod tests {
    use kasane_plugin_sdk::test::TestHarness;
    use crate::exports::kasane::plugin::plugin_api::Guest;

    #[test]
    fn highlights_cursor_line() {
        let mut h = TestHarness::new();
        h.set_cursor_line(7);
        let _ = crate::__KasanePlugin::on_state_changed_effects(crate::dirty::ALL);
        let out = crate::__KasanePlugin::display();
        assert_eq!(out.len(), 1);
    }
}
```

Full cookbook in [`docs/plugin-testing.md`](../plugin-testing.md). When
`test-harness` is active on a non-wasm target, the macro-emitted
`host_state::*` calls route into the harness; production WASM builds
are unchanged.

### 9.2 `kak_lint!` compile-time command validator ([Issue #93](https://github.com/Yus314/kasane/issues/93))

Wrap raw Kakoune command literals in `kak_lint!` to catch typo'd or
unknown flags at compile time:

```rust
use kasane_plugin_sdk::kak_lint;

let ok = kak_lint!("declare-user-mode 'sprout'");
// Compile error: unknown flag `-override` for `declare-user-mode`.
// let bad = kak_lint!("declare-user-mode -override 'sprout'");
```

The catalog covers 12 setup-heavy commands (`declare-user-mode`,
`define-command`, `map`, `declare-option`, `set-option`,
`unset-option`, `evaluate-commands`, `hook`, `alias`, `echo`, `info`,
`try`). Unknown commands pass through unchanged — additive policy.

### 9.3 Structured `KakCommand` enum ([ADR-043](../decisions.md#adr-043-structured-kakcommand-enum-for-type-safe-kakoune-command-construction))

For plugins that compose Kakoune commands programmatically (build a
body from user input, inspect a command before issuing, etc.), the
`kasane_plugin_sdk::kak_cmd` module exposes a `KakCommand` enum with
per-variant builders:

```rust
use kasane_plugin_sdk::kak_cmd::{KakCommand, DeclareUserMode, DefineCommand, Map, Scope};

let setup: Vec<KakCommand> = vec![
    DeclareUserMode::new("sprout").try_idempotent().into(),
    DefineCommand::new("bump", "increment-counter")
        .override_existing()
        .docstring("bump the sprout counter")
        .into(),
    Map::new(Scope::Global, "sprout", "b", ":bump<ret>")
        .docstring("bump")
        .into(),
];
let strings: Vec<String> = setup.iter().map(KakCommand::render).collect();
```

The renderer cannot produce an unknown flag — builders only expose
flags Kakoune accepts (e.g. `DeclareUserMode` does not expose an
`override_existing()` method because Kakoune does not accept
`-override` there).

### 9.4 `kak::set_option` / `kak::set_option_add` ([Issue #97](https://github.com/Yus314/kasane/issues/97))

Two new helpers in the existing `kak::*` module defuse the bareword
`%X{…}` expansion foot-gun: Kakoune does not expand expansions when
they're glued to a bareword, only inside double-quoted strings.

```rust
use kasane_plugin_sdk::kak::{self, Scope};

// Each entry is wrapped in `"..."` so `%opt{…}` / `%arg{…}` expand
// at command-evaluation time. Writing this by hand as a bareword
// would store the literal text — silent failure mode.
let cmd = kak::set_option_add(Scope::Window, "ui_options", &[
    ("sprout_request_id", "%opt{sprout_request_seq}"),
    ("sprout_request_kind", "pick"),
    ("sprout_request_arg", "%arg{1}"),
]);
```

Use `set_option` / `set_option_add` for any option value containing
a `%X{…}` expansion. For literal values where expansions should not
fire, the existing `escape_arg` (single-quoted) remains correct.

### 9.5 Choosing between the three command-construction layers

| Layer | Best for |
|---|---|
| `kak::*` helpers | one-liner string, no composition |
| `kak_lint!` macro | hand-composed string literals that need flag validation |
| `kak_cmd::KakCommand` | programmatic composition, inspection, transformation |

All three layers coexist; pick based on the call site's needs.

## 9.1 `PluginBackend` trait removed

The internal `PluginBackend` trait, the `IsBridgedPlugin` marker, and
the `kasane_core::plugin::SlotImpl` enum are deleted in 0.7. The only
native plugin authoring surface is `Plugin` + `HandlerRegistry`, and
the framework wraps each `Plugin` impl in a `PluginBridge`
internally.

If your code references any of these, the fix shape is:

| Old (0.6) | New (0.7) |
|---|---|
| `impl PluginBackend for MyPlugin { ... }` | `impl Plugin for MyPlugin { type State = …; fn id(&self) -> PluginId { … } fn register(&self, r: &mut HandlerRegistry<Self::State>) { … } }` — see [plugin-development.md Quick Start](../plugin-development.md#quick-start) |
| `Box<dyn PluginBackend>` field | `Box<PluginBridge>` (concrete; the dispatch surface is now inherent on `PluginBridge`) |
| `PluginRuntime::register_backend(Box::new(MyPlugin))` | `PluginRuntime::register(MyPlugin)` (handles the `PluginBridge::new` wrap internally) |
| `is_bridge()` discriminator | gone — every registered plugin is a `PluginBridge` |
| `PluginFactory::create() -> Box<dyn PluginBackend>` | `PluginFactory::create() -> Box<PluginBridge>`; `host_plugin` / `builtin_plugin` closures must return `PluginBridge` (typically `PluginBridge::new(MyPlugin)`) |

The `kasane_wasm::WasmPlugin` re-export now points at `PluginBridge`
(it was the loader's owned adapter type in 0.6); WASM plugin call
sites that worked with `WasmPlugin` continue to compile because every
PluginBackend method has an inherent equivalent on `PluginBridge`.

## 10. Things that did **not** change

The following surfaces are stable across 0.6 → 0.7:

- `Element` non-text variants (`Container`, `StyledLine`, `Image`, etc.).
- `KeyMap` / `ChordBinding` DSL.
- KDL configuration (`kasane.kdl`) — no schema changes.
- Theme tokens and their resolution rules.
- `Lens` trait public surface.
- `kasane-plugin-model`, `kasane-plugin-package`.

If a 0.6 plugin only uses these stable surfaces and does not touch any
of §§1–8, it builds against 0.7 with no changes.

## 11. Reporting issues

If a migration site is not covered here, please file a bug at
<https://github.com/Yus314/kasane/issues> with:

- The plugin's `Cargo.toml` `dependencies` section.
- The full compile error from `cargo build` against the 0.7 SDK.
- A minimal reproduction (a single `Atom`/`Element`/`DisplayDirective`
  construction site, ideally).

The R2.x P7+P9 cascade ([ADR-039](../decisions.md#adr-039-plugin-path-consolidation-r2x))
is the canonical reference for the rationale; the 8-PR commit chain
landed on 2026-05-08–10 and is queryable via
`git log --grep "R2.x P7"`.
