# Migration Guide — Kasane 0.5 → 0.6

This guide covers every breaking change between Kasane 0.5.0 and 0.6.0. Read it
end-to-end if you maintain a plugin; skim it if you only configure Kasane.

The 0.6.0 release ships **three coordinated WIT ABI bumps** (1.0.0 → 2.0.0 →
3.0.0), the consolidation of the plugin authoring path on `Plugin +
HandlerRegistry` (ADR-038), and the retirement of several legacy types and
function names that had been deprecated through 0.5.x.

> **TL;DR for plugin authors**
> 1. Pin `kasane-plugin-sdk = "0.6"` (and `kasane-plugin-sdk-macros = "0.6"` if
>    used directly), then `cargo build` your plugin against the regenerated WIT.
> 2. Rename `annotate_*` handlers to `decorate_*`.
> 3. Replace any `Transparent*` types with `KakouneSafe*`.
> 4. Drop `Hash` derives from your `Plugin::State` if you only had them to satisfy
>    the trait — the bound is gone.
> 5. If you implement `PluginBackend` directly (rather than `Plugin +
>    HandlerRegistry`), migrate to the public path. ADR-038 makes
>    `PluginBackend` internal.
> 6. If you read selections via the legacy `selection` record or the
>    `get-selection*` triplet, switch to the WIT 3.0 `selection-set` value-record
>    and the new history accessors.

---

## 1. WIT contract — 0.x → 3.0.0

The wire contract changed in three steps within the 0.6.0 development cycle.
Out-of-tree plugins must rebuild against the 0.6.0 SDK; binaries built against
the 0.5-era WIT are rejected at load time by the host's version check.

### 1.1 WIT 1.0.0 — `brush` / `style` / `inline-box` (ADR-031 Phase 4)

The legacy `face` record + attribute bitflags are replaced by:

| Removed (0.5)            | Replacement (0.6)                                   |
|--------------------------|-----------------------------------------------------|
| `face` record            | `style` record (+ `brush` colour variant)           |
| `attribute` bitflags     | Explicit `style` fields (`bold`, `underline`, ...)  |
| Per-call colour records  | `brush` variant (RGB, named theme token, gradient)  |

Inline-box host paint became a first-class extension point. Plugins that emit
inline-box directives now also export a `paint-inline-box(box-id)` callback to
produce the rendered cells; the host enforces a recursion-depth ≤ 8 + cycle
detection guard.

### 1.2 WIT 2.0.0 — Style-native function names (ADR-031 closure)

Host-side functions that previously took or returned `face` records are renamed
to their Style-native counterparts. The mechanical changes:

| 0.5 / WIT 1.0          | 0.6 / WIT 2.0                |
|------------------------|------------------------------|
| `get-menu-style` (string return — name collision) | `get-menu-mode` (string) |
| Various `*-face` host-state getters / setters | `*-style` |
| `style::from-face` / `to-face` (round-trip helpers) | retained as `#[doc(hidden)]` wire-parser helpers; do not depend on them |

The `face`, `color`, and `attribute` types remain in the WIT for parser
compatibility but are `#[doc(hidden)]`. Treat them as removed for new code.

### 1.3 WIT 3.0.0 — Selection / Time / History (ADR-035)

ADR-035 lands first-class selection sets and time-travel history queries.

**Removed**:

```wit
// WIT 2.x — gone in 3.0
record selection { line: u32, column: u32, ... }
get-selection() -> selection;
get-selection-line() -> u32;
get-selection-column() -> u32;
```

**Added**:

```wit
// WIT 3.0
record selection-set {
    primary: u32,                       // index into selections
    selections: list<selection-record>, // disjoint, sorted, non-empty
    direction: selection-direction,
}

variant time {
    now,
    at(version-id),
}

type version-id = u64;

// New history accessor functions (host-side)
text-at-time: func(time: time) -> result<list<string>, error>;
selection-at-time: func(time: time) -> result<selection-set, error>;
display-directives-at-time: func(time: time) -> result<list<display-directive>, error>;
current-version: func() -> version-id;
```

If your plugin used `get-selection()` to read a single primary selection, the
direct replacement is reading `primary` out of `selection-set` and indexing
into `selections`. If you only need the head/anchor of the primary selection,
the `selection-record` shape exposes both endpoints with explicit `direction`.

The `selection-set-to-kakoune-command` helper provides the canonical projection
back to a Kakoune `:select` argument string.

> **ADR-034 wire-level collapse — deferred**: A coordinated 3.0 design also
> proposed collapsing `display-directive` into a unified `display` variant,
> driven by ADR-034. This was reconsidered post-implementation review and is
> deferred indefinitely. WIT 3.0 ships the ADR-035 changes only.

### 1.4 Bundled plugin rebuild

The host enforces an exact major-version match against the SDK at load time.
After bumping `kasane-plugin-sdk = "0.6"`:

```bash
cd /path/to/your/plugin
cargo component build --release
# verify
wasm-tools component wit target/wasm32-wasip2/release/your_plugin.wasm | head
```

The bundled examples under `examples/wasm/*` are regenerated as part of the
0.6.0 release and serve as up-to-date reference implementations.

## 2. Handler renames — `annotate_*` → `decorate_*`

The four annotation handler families are renamed for terminology consistency
with the existing `decorate_cells()` / `RenderOrnaments` surface.

| 0.5                       | 0.6                       |
|---------------------------|---------------------------|
| `annotate_gutter`         | `decorate_gutter`         |
| `annotate_background`     | `decorate_background`     |
| `annotate_inline`         | `decorate_inline`         |
| `annotate_virtual_text`   | `decorate_virtual_text`   |

`HandlerRegistry::on_annotate_*` registrations become
`HandlerRegistry::on_decorate_*`. The handler signatures and semantics are
unchanged — this is a pure rename.

For SDK-macro users (`define_plugin!`), update each section:

```diff
 define_plugin! {
-    annotate_gutter(line) { ... }
+    decorate_gutter(line) { ... }
-    annotate_background(line) { ... }
+    decorate_background(line) { ... }
 }
```

## 3. Type renames — `Transparent*` → `KakouneSafe*`

The transparency / Level-5 footprint types in `kasane-core::plugin` are renamed
to `KakouneSafe*` to make the contract explicit (the type asserts that an
effect is safe to forward to Kakoune unchanged).

| 0.5                              | 0.6                              |
|----------------------------------|----------------------------------|
| `TransparentCommand`             | `KakouneSafeCommand`             |
| `TransparentEffects`             | `KakouneSafeEffects`             |
| `TransparentDirective`           | `KakouneSafeDirective`           |
| `Transparent*` accessor methods  | `KakouneSafe*` accessors         |

All public re-exports follow the rename. `plugin_prelude` exports the new
names; the old names are not re-exported under aliases.

---

## 4. `Hash` bound dropped from `PluginState`

`Plugin::State` previously required `Clone + PartialEq + Debug + Send + Hash`.
The `Hash` requirement is gone — change detection between frames now uses
`dyn_eq` instead of pre-hashing.

Implications:

- If you only added `#[derive(Hash)]` to satisfy the trait bound, you can drop it.
- If your state contains types that cannot implement `Hash` (e.g. floats,
  ordered maps with non-`Hash` keys), this constraint is lifted.
- If your state uses interior mutability, ensure `dyn_eq` produces stable
  results across frames; the change-detection pass relies on it.

No code change is required for plugins whose state already implemented
`PartialEq` correctly.

---

## 5. Plugin authoring path consolidation (ADR-038)

`Plugin + HandlerRegistry` (introduced in 0.3.0 under ADR-025) is now the
**sole** sanctioned authoring path. `PluginBackend` is the internal dispatch
ABI consumed by `PluginRuntime` and `WasmPlugin`; new extension points are
added via `HandlerRegistry::on_X(...)` registrations, not via new
`PluginBackend` methods.

If your plugin currently `impl`s `PluginBackend` directly (this was possible in
0.5.x but never the recommended path):

```diff
-impl PluginBackend for MyPlugin {
-    fn render_widget(&self, ctx: &Ctx) -> Element { /* ... */ }
-    fn on_key(&self, ctx: &Ctx, key: Key) -> Effects { /* ... */ }
-}
+impl Plugin for MyPlugin {
+    type State = MyState;
+
+    fn register(reg: &mut HandlerRegistry<Self>) {
+        reg.on_render_widget(|state, ctx| { /* ... */ });
+        reg.on_key(|state, ctx, key| { /* ... */ });
+    }
+
+    fn initial_state(&self) -> Self::State { MyState::default() }
+}
```

The capability-trait reorganisation (R1.x) means the previously-separate
traits (`Lifecycle`, `Io`, `PubSubMember`, `ExtensionParticipant`, ...) are now
super-traits of `PluginBackend`. They are blanket-implemented for any
`Plugin + HandlerRegistry` user automatically; you do not need to implement
them individually.

The `BuiltinDragPlugin`, `BuiltinFoldPlugin`, `BuiltinMouseFallbackPlugin`,
`BuiltinInputPlugin`, `BuiltinShadowCursorPlugin`, `BuiltinMenuPlugin`,
`BuiltinInfoPlugin`, `BuiltinDiagnosticsPlugin`, and `ProjectionStatusPlugin`
in-tree migrate to `Plugin + HandlerRegistry` opportunistically when adjacent
work touches them — there is no coordinated push, and external authors are
expected to migrate at their own pace within the 0.6.x line.

## 6. `Command::Update*` removal — `StateUpdates` channel

The `Command::Update*` variants and the `extract_*_update` helpers are
replaced by a typed `StateUpdates` channel on `Effects`. The two surfaces had
overlapping responsibilities for emitting state-shape updates back to the host;
`StateUpdates` is the single coherent path.

Migrate in two mechanical steps:

```diff
 fn handler(&self, state: &State, ctx: &Ctx) -> Effects {
     let mut effects = Effects::default();
-    effects.commands.push(Command::UpdateBuffer(buf_id, new_text));
+    effects.state_updates.update_buffer(buf_id, new_text);
     effects
 }
```

The trybuild fixture `state_updates` in `kasane-macros` is the canonical
template for SDK-macro users.

## 7. SDK version pinning

Pin both SDK crates to `0.6`:

```toml
[dependencies]
kasane-plugin-sdk = "0.6"
kasane-plugin-sdk-macros = "0.6"  # only if used directly
```

The host enforces an exact major-version compatibility check at WASM load
time; mixing `0.5` SDK output with a `0.6` host will fail with a clear
`PluginVersionMismatch` diagnostic.

## 8. Things that did **not** change

The following surfaces are stable across 0.5 → 0.6:

- `Element` / `Atom` / `BufferLine` shape and field semantics
- `KeyMap` / `ChordBinding` DSL
- KDL configuration (`kasane.kdl`) — no schema changes
- Theme tokens and their resolution rules
- `Lens` trait public surface (the *additions* — caching, registration — are
  additive; existing lenses continue to work)
- `kasane-plugin-model` (`PluginId`, `SettingValue`, etc.) — at 0.1.0,
  unaffected by the host version bump
- `kasane-plugin-package` (`.kpk` build / inspect / verify) — at 0.1.0,
  unaffected

If a 0.5 plugin only depends on the surfaces above and does not touch any of
the breaking surfaces in §§1–6, it builds against 0.6 with at most a
`Cargo.toml` version pin.

## 9. Reporting issues

If you hit a migration issue not covered here, please file a bug at
<https://github.com/Yus314/kasane/issues> with:

- The plugin's `Cargo.toml` `dependencies` section
- The exact compiler / `cargo component` error
- The kasane host version (`kasane --version`) and SDK version

Migration regressions are treated as 0.6.x patch-blocking.
